# 二进制与部署布局（激进版）

## 1. 目标

本规范直接定义 Honeycomb 的新部署结构，不保留旧模型兼容：

- 三个主二进制
- 明确运行时智能执行与长期能力治理分离
- 明确 LLM 在蜂执行链路中的核心地位

## 2. 主二进制清单（固定）

1. `honeycomb-bee`：蜂运行时，承担 queen/worker 执行
2. `honeycomb`：蜂巢能力中心，承担全局能力资产管理
3. `honeycomb-evolution`：生态进化中心，承担全量观测与治理优化

不再延续“`honeycomb` 同时承担所有执行与控制”的做法。

## 3. 二进制职责边界

### 3.1 `honeycomb-bee`

负责：

- queen/worker 生命周期管理
- LLM 驱动的任务内决策与执行
- 职责提示词文档加载、工具调用与证据记录
- 任务上下文、会话态、局部记忆管理

禁止：

- 直接写长期能力注册表
- 直接发布全局最佳实践
- 直接修改系统默认实现策略

### 3.2 `honeycomb`

负责：

- skill、tool、implementation、template、practice 注册与查询
- 全局工具白名单与能力约束管理
- 对 bee 暴露版本化能力视图

禁止：

- 承担蜂后/工蜂的实时执行循环

### 3.3 `honeycomb-evolution`

负责：

- 汇总跨任务全量观测
- 评分、谱系、晋升淘汰与实践沉淀
- 输出并执行治理决策

禁止：

- 直接参与任务内实时派发

## 4. LLM 执行约束

- bee 主执行链必须声明并加载 LLM 实现体
- 职责提示词文档是正式输入资产，必须版本化
- 工具调用必须经过能力中心下发的受控清单

## 5. 记忆与上下文布局

- bee：任务/会话级上下文与短期记忆
- honeycomb + evolution：长期记忆、最佳实现、全局策略

禁止将任务内临时状态直接提升为长期系统默认。

## 6. 部署建议

### 6.1 单机最小部署

- 常驻 `honeycomb-bee`
- 常驻 `honeycomb`
- 按需或常驻 `honeycomb-evolution`

### 6.2 分层部署

- bee 可按任务规模水平扩展
- honeycomb 保持稳定能力服务
- evolution 独立扩容做离线/准实时分析

## 7. 当前结论

1. Honeycomb 主二进制从 2 个升级为 3 个并固定。
2. 执行智能统一收敛到 `honeycomb-bee`。
3. 能力中心与进化中心承担长期状态与治理，不再混入运行时执行。
