# 二进制与部署布局

## 1. 目标

本文档定义 Honeycomb 当前阶段应保持多少个主二进制、各自承担什么职责、如何补强功能，以及在什么条件下才考虑新增独立控制面二进制。

## 2. 当前确认

当前建议保持两个主二进制：

- `honeycomb`
- `honeycomb-evolution`

这样做的核心原因不是“以后永远只有两个”，而是当前阶段要先稳住执行面和进化面的职责边界，避免在实现尚未成形时过早把模块问题升级成进程和部署问题。

## 3. 为什么当前先保持两个

### 3.1 执行面与进化面天然是两类问题

执行面关注：

- 当前任务怎么跑
- 工蜂怎么连回蜂后
- 当前状态怎么落盘
- 当前失败怎么恢复

进化面关注：

- 跨任务结果怎么汇总
- 哪些实现体应晋升或淘汰
- 最佳实践如何沉淀
- 长期默认值如何更新

这两类问题在时间尺度、状态模型和权限边界上都明显不同，因此保持两个主二进制是合理的最小拆分。

### 3.2 现在再加更多主二进制成本高于收益

如果当前就新增：

- `honeycomb-ctl`
- `honeycomb-server`
- `honeycomb-observe`

会带来：

- CLI 和服务边界过早冻结
- 部署路径提前复杂化
- 模块还没稳定就先拆远程接口
- 审计、权限、配置会被分散到更多入口

因此现阶段更适合：

- 主二进制数量保持 2
- 通过子命令和模块补强能力
- 等控制面需求真正稳定后再决定是否拆第 3 个二进制

## 4. `honeycomb` 的职责与补强建议

### 4.1 核心职责

`honeycomb` 负责：

- 任务创建与启动
- 蜂后与工蜂角色运行
- 任务拓扑推进
- assignment 派发与结果收集
- 运行态落盘
- 事件、追踪、审计记录
- 触发器与常驻蜂巢的执行侧能力

### 4.2 当前建议补强的能力

建议不新增新二进制，而是在 `honeycomb` 内补强以下入口：

- `queen run`
- `worker run`
- `task submit`
- `task inspect`
- `task replay`
- `trigger create`
- `trigger list`
- `trigger fire`
- `resident run`
- `audit tail`

这些入口的意义是让执行面同时具备：

- 运行入口
- 查询入口
- 调试入口
- 最小控制入口

### 4.3 控制能力如何放进去

当前阶段建议把控制能力做成：

- `honeycomb` 的子命令
- 或 `honeycomb` 内部可选本地 API 模式

而不是独立控制面二进制。

原因：

- 执行面还在定义最小闭环
- 查询、调试、回放和审批触发都强依赖本地文件布局
- 过早拆控制面会迫使我们过早固定远程协议

### 4.4 执行器能力也应在 `honeycomb` 内补强

建议明确 `honeycomb` 包含执行器适配层：

- 本地进程执行
- 脚本执行
- 未来沙箱执行

第一版只需稳定本地进程执行，但结构上要给更多执行方式留位置。

## 5. `honeycomb-evolution` 的职责与补强建议

### 5.1 核心职责

`honeycomb-evolution` 负责：

- 读取评估输入
- 生成评分报告
- 做治理决策
- 更新谱系和最佳实践
- 通过正式路径更新注册表

### 5.2 当前建议补强的能力

建议至少支持两种运行模式：

- `batch run`：批处理历史评估输入
- `watch ingest`：持续发现新评估输入并增量处理

建议补充的子命令方向：

- `fitness run`
- `fitness explain`
- `governance plan`
- `governance apply`
- `registry sync`
- `lineage show`
- `practice publish`

### 5.3 审批与回滚能力应内建

进化面不能只会“算分”，还应负责：

- 产出待审批变更
- 解释为什么建议晋升或淘汰
- 生成可回滚的注册表更新
- 回指相关执行证据

也就是说，治理闭环不应落到外部脚本里拼接。

## 6. 何时考虑新增第 3 个主二进制

只有当以下条件同时变强时，才建议评估新增独立控制面二进制，例如 `honeycomb-ctl`：

- 查询、回放、审批已经成为高频独立使用场景
- 控制面需要独立部署，而不是附着在执行或进化进程内
- UI、API、CLI 三者都需要共享稳定控制协议
- 审计和权限检查已经形成统一控制面模型

适合新增的第 3 个二进制应是：

- 只负责管理、查询、审计、审批、回放
- 不直接承担任务执行
- 不直接承担评分与治理计算

不建议优先新增的类型：

- 独立观察二进制
- 独立触发器二进制
- 独立调试二进制

这些能力当前都更适合保留在已有两个主二进制内。

## 7. 部署布局建议

### 7.1 第一版本地部署

建议支持：

- 单机只运行 `honeycomb`
- 按需离线运行 `honeycomb-evolution`

这是最小验证路径。

### 7.2 增强型部署

后续可支持：

- 常驻 `honeycomb`
- 定期运行或常驻 `honeycomb-evolution`

此时两者通过文件或正式数据接口交换。

### 7.3 暂不建议的部署

当前不建议优先承诺：

- 多种独立服务同时常驻
- 执行面、进化面、控制面全部拆成独立守护进程
- 一开始就做远程服务编排集群

## 8. 设计原则

- 主二进制数量先少后多
- 功能扩展优先走子命令和模块，不优先走新进程
- 新增二进制必须解决明确的部署问题，而不是掩盖模块边界问题
- 控制面独立必须晚于执行面和进化面闭环稳定

## 9. 当前结论

当前阶段建议明确：

1. 主二进制保持 2 个不变。
2. `honeycomb` 补强运行、查询、调试、触发、常驻能力。
3. `honeycomb-evolution` 补强批处理、增量摄取、审批、回滚和解释能力。
4. 第 3 个控制面二进制只做预留，不在当前阶段落地。

## 10. 总结

Honeycomb 当前最重要的不是“拆出更多名字不同的程序”，而是让执行面和进化面各自先长成完整、可运行、可治理的系统。只要这两个主二进制先站稳，后续是否增加控制面二进制就会是顺势而为，而不是返工。
