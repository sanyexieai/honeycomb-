# 工蜂物理复制与生命周期管理

## 1. 目标

本规范定义 Honeycomb 如何把 `honeycomb-bee` 实例化为多个工蜂，以及这些工蜂从生成到退出的完整生命周期。

这里的重点是把“逻辑上同构”和“物理上可复制”区分开来，并且给后续实现一个稳定的物理模型。

## 2. 两种模式

### 2.1 逻辑复制

逻辑复制表示：

- 不真的复制二进制文件
- 由同一路径下的 `honeycomb-bee` 以不同参数启动
- 通过角色、任务、节点身份来区分实例

优点：

- 简单
- 启动快
- 便于调试

缺点：

- 物理隔离较弱
- 不适合做固定快照型运行实例

### 2.2 物理复制

物理复制表示：

- 蜂后把当前 `honeycomb-bee` 复制到某个工蜂目录
- 再从该目录启动工蜂实例
- 工蜂在该目录中拥有自己的本地元数据和日志

优点：

- 物理实体清晰
- 便于快照、归档、排查和回放
- 更符合蜂巢单元的独立性

缺点：

- 启动成本更高
- 文件管理更复杂
- Windows 下二进制替换要谨慎处理

## 3. 当前建议

建议采用分阶段路线：

- 第一版先支持逻辑复制
- 同时把物理复制的目录结构、元数据和生命周期设计好
- 第二阶段再真正启用物理复制

这样可以避免一上来把平台相关细节和核心执行协议绑死，同时不影响 LLM 驱动执行主链路。

## 4. 工蜂物化目录建议

当启用物理复制时，建议目录结构如下：

```text
runtime/
  tenant/
    <tenant_id>/
      namespaces/
        user/
          <user_id>/
            tasks/
              <task_id>/
                workers/
                  <worker_id>/
                    honeycomb-bee.exe
                    node.json
                    bootstrap.json
                    logs.jsonl
                    events.jsonl
```

说明：

- `honeycomb-bee.exe`：被物理复制出的工蜂二进制
- `node.json`：节点身份元数据
- `bootstrap.json`：启动参数与父级信息
- `logs.jsonl`：本地运行日志
- `events.jsonl`：工蜂局部事件流

## 5. 物化流程建议

建议最小流程：

1. 蜂后决定需要新工蜂
2. 蜂后生成 `worker_id`
3. 创建工蜂目录
4. 复制二进制到工蜂目录
5. 写入 `bootstrap.json` 和 `node.json`
6. 启动该目录下的 `honeycomb-bee`
7. 工蜂向蜂后发起 `hello`
8. 蜂后确认并纳入任务运行态

## 6. 启动参数建议

无论逻辑复制还是物理复制，建议都通过明确参数启动：

- `--role worker`
- `--task-id <task_id>`
- `--node-id <worker_id>`
- `--queen-node-id <queen_id>`
- `--tenant-id <tenant_id>`
- `--namespace <namespace>`
- `--hive-id <hive_id>`
- `--queen-addr <addr>`
- `--queen-token <token>`
- `--worker-dir <path>`
- `--prompt-bundle-ref <ref>`
- `--llm-provider <provider>`

这样可以保证物理复制和逻辑复制共用同一套启动语义，并保证 worker 启动时具备可追溯的 LLM 执行配置。

## 7. 生命周期阶段

建议至少划分这些阶段：

- `planned`：已决定需要工蜂，但尚未创建
- `materialized`：已创建目录与元数据
- `spawned`：已启动进程
- `connected`：已完成握手
- `idle`：空闲等待 assignment
- `running`：正在执行 assignment
- `reporting`：正在回传结果或结束状态
- `terminating`：收到关闭指令，准备退出
- `terminated`：已退出
- `failed`：启动失败或运行中断

## 8. 生命周期管理职责

### 8.1 蜂后职责

蜂后负责：

- 决定是否需要工蜂
- 创建或注册工蜂
- 维护工蜂状态
- 派发 assignment
- 判断工蜂是否应被关闭、替换或回收
- 维护工蜂提示词与能力引用版本一致性

### 8.2 工蜂职责

工蜂负责：

- 读取启动信息
- 注册自己
- 执行 assignment（LLM + 职责提示词文档 + 工具）
- 回传状态与结果
- 优雅退出

## 9. 关闭与回收建议

任务结束后，建议蜂后执行：

1. 停止派发新 assignment
2. 等待正在运行的 assignment 结束或超时
3. 向存活工蜂发送 `shutdown`
4. 更新工蜂状态为 `terminating`
5. 等待退出
6. 标记为 `terminated`

第一版不建议自动删除工蜂目录，可先保留用于排查和回放。

## 10. 失败场景

需要重点考虑：

- 二进制复制失败
- 工蜂目录创建失败
- 启动后无法连接蜂后
- 握手失败
- 工蜂运行中崩溃
- 工蜂关闭超时

这些状态都应进入：

- `events.jsonl`
- `trace.jsonl`
- `audit.jsonl`

## 11. 与资源控制的关系

物理复制不应绕过资源与配额限制。

也就是说：

- 先通过资源与配额校验
- 再创建工蜂目录或启动工蜂
- 不允许先复制一批工蜂再决定是否要用

## 12. 与安全模型的关系

物理复制不改变信任边界。

即使工蜂拥有独立目录，也仍然必须：

- 校验 `tenant_id`
- 校验 `task_id`
- 校验 `namespace`
- 校验 `queen_token`
- 只连接所属蜂后
- 只加载能力中心允许的工具与实现引用

## 13. 与回放和审计的关系

保留物理工蜂目录的好处是：

- 可以查看当时的工蜂元数据
- 可以查看局部日志与局部事件
- 可以为任务级回放补充工蜂视角证据

因此，第一版即使不做自动清理，也有现实价值。

## 14. 第一版建议

第一版建议：

- 执行逻辑以逻辑复制为主
- 生命周期模型按物理复制设计
- 保留 `worker_dir` 和 `bootstrap.json` 结构位
- 让后续启用物理复制时不需要推翻协议、状态机和目录布局

## 15. 总结

工蜂物理复制的关键不是“复制文件”本身，而是：

- 如何实例化节点身份
- 如何管理生命周期
- 如何记录证据
- 如何与任务、配额、安全和恢复对齐

只要这几个点定稳，逻辑复制升级为物理复制就会自然很多。
