# 记忆、上下文与 LLM 角色（目标与现状）

## 0. 与当前架构的关系

正式架构以三二进制、三层职责为准：`honeycomb-bee`（bee 运行时）、`honeycomb`（蜂巢能力中心）、`honeycomb-evolution`（进化面）。详见 `architecture.md`、`bee-runtime-hive-ecology-architecture.md`。下文「现状」若与代码不同步，以仓库实现为准；目标行为以 bee 主链 **LLM 驱动** 为准。

## 1. 为什么需要本文档

架构文档中「蜂后解释任务拓扑、派发工蜂」描述的是**目标职责模型**；若读者默认**当前二进制已具备**自然语言式「理解」或已内建 LLM 推理，会产生与实现对齐的误解。

本文档说明：

- **记忆与上下文**建议落在文档中的分层（任务/会话 vs 生态/进化）
- **没有 LLM 时**，当前执行路径里「谁」在推进任务、与蜂后文档职责的**差距**
- **「核心蜂巢由 LLM 驱动」**若作为产品原则，应落在**哪一层**才与仓库现状一致

## 2. 记忆与上下文的分工（建议）

### 2.1 上下文（context）

- **归属**：以**任务与会话**为第一落点（见 `task-runtime.md`：任务运行时、任务蜂巢会话、`local_state` 等）。
- **含义**：本次运行中可被追溯、重放、审计的输入、状态与中间产物；工蜂与蜂后协议中携带的 `task_id` / `tenant_id` / `namespace` 等也属于运行上下文。
- **原则**：多节点协作、重放与审计要求上下文必须**结构化落盘**，不能等同于「模型窗口里的隐式记忆」。

### 2.2 记忆（memory）

建议拆成两类，避免混在单一「蜂巢最小单位」里一锅煮：

| 类型 | 含义 | 建议落点 |
| --- | --- | --- |
| **任务/会话级** | 本次任务内需要保留的中间结论、引用、槽位状态 | 任务运行时、会话、`state.slots`（见 `hive-spec.md`）、事件与 trace |
| **长期/跨任务** | 组织知识、可演化最佳实践、跨任务检索结果策略 | 生态层技能包、外部存储、注册表、practice、进化与治理流程 |

**结论**：**上下文**主要在**最小运行单位（任务/会话）**实现；**长期记忆的策略与内容**主要在**生态与治理**扩展；最小蜂巢通过**契约与工具引用**接入，而不是在每个蜂巢内复制一套向量库或记忆引擎。

## 3. 现状：没有 LLM 时，蜂后如何「理解拓扑、派发工作」？

### 3.1 文档中的蜂后（目标）

`queen-worker-model.md` 中蜂后职责包括：解释任务拓扑、派生工蜂、下发派发单元、汇总结果等。这是**角色与协议层面的设计方向**。

### 3.2 当前实现中的蜂后进程

当前 `honeycomb queen run` **不**调用语言模型，也**不**解析任务图或自动生成派发计划；其行为是输出蜂后 endpoint、协议版本与 token 等**就绪信息**（协议/演示骨架），而非「认知式」调度。

因此：**在现有代码路径下，并不存在「蜂后用模型理解拓扑」这一步**；若未引入其它组件，「拓扑如何被满足」来自**确定性机制**与**显式操作**，例如：

- **调度器**（`scheduler run-once` / `run-loop`）：按规则扫描排队任务、触发器就绪条件等，创建 **assignment**，并可将任务推进到完成（见调度模块与测试中的行为）。
- **显式派发**：如 `task assign` 等路径写入派发单元，由工蜂侧执行。
- **演示/串联流程**：如 `task demo-flow` 用于最小闭环演示。

以上路径均为**代码与数据驱动**，不是蜂后进程内的 LLM 推理。

### 3.3 LLM 在当前仓库中的实际位置

`implementation-spec.md` 中的「最小 LLM 接入约定」表明：**模型调用**当前主要通过 **skill 实现体**（如 `skill execute` 与具体 `executor`）发起，即**工蜂/技能执行路径**可加载实现体并调用 provider，而不是 `queen run` 内嵌推理。

### 3.4 与「核心蜂巢由 LLM 驱动」的重构后表述

按当前激进重构方向，正式表述改为：

- **bee 运行时主链路必须 LLM 驱动**，queen/worker 的执行决策不再以纯规则分支替代。
- **职责提示词文档**是可版本化执行资产，必须参与实际运行，而非仅作说明文字。
- **蜂后理解拓扑并派发**应落在 bee 的正式实现体中，成为默认能力，不再视为可选增量。
- **任务内短期记忆与上下文**由 bee 承担，**长期记忆与最佳实现**由 hive/evolution 承担并治理。

## 4. 文档与实现的对齐建议

- 阅读 `architecture.md` / `queen-worker-model.md` 时，将「蜂后解释拓扑」理解为**目标职责**；核对**具体子命令**是否已实现对应逻辑。
- 讨论「LLM 驱动」时，指明是 **skill/实现体层**、**未来蜂后策略层**，还是 **调度/规则层**，避免混为一谈。
