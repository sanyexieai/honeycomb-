# 设计状态矩阵

本文档用于记录 Honeycomb 各项能力当前的设计状态。

状态说明：

- `已设计`：已有独立文档，边界基本明确
- `部分设计`：方向已定，但仍需补细节
- `未设计`：已确认重要，但尚无正式文档
- `暂不实现但需预留`：短期不做，但必须预留结构位置

- 总体架构与双平面模型
- 领域边界：任务、角色、技能、进化
- 蜂后与工蜂模型
- 节点身份模型
- 蜂后与工蜂协议
- 协议示例文档
- 任务拓扑
- 任务运行时
- 任务调度基础模型
- 运行时事件模型
- 运行时存储布局
- 多用户模型
- 多租户模型
- 权限与可见性
- 资源控制与配额
- 故障恢复基础模型
- 安全与信任模型
- 审计与追踪基础模型
- 可观测性与回放基础模型
- 执行面与进化面的数据样例
- 蜂巢规范、实现体规范、最佳实践规范
- 进化系统、评分与晋升基础模型
- 技能注册与发布
- 技能包格式与导入导出
- 工蜂物理复制与生命周期模型
- 调试与查看工具形态
- 进化面与注册表联动
- 任务模板与编排模板
- 远程技能仓库与分发模型
- 模板与最佳实践联动
- 常驻蜂巢模型
- 生态层组合蜂巢模型
- 触发器与定时调度模型
- 调度状态与触发历史模型
- 执行面最小实现路线图
- 进化面最小实现路线图

## 2. 部分设计

- 技能包格式的压缩封装与远程分发细节
- 调试工具的具体交互形态
- 自动模板生成与推荐

## 3. 暂不实现但需预留

- 跨租户共享
- 自动蜂后接管
- 分布式一致性
- 高级公平调度
- 工蜂池复用
- 图形化控制台
- 复杂任务模板市场
- 自动模板生成
- 多仓库聚合

## 4. 当前最值得继续补的文档

1. 技能包压缩封装与校验细则
2. 执行面代码模块映射草案
3. 进化面代码模块映射草案

## 5. 设计原则

- 先把边界定对，再做实现
- 能暂时不做，但不能没有位置
- 数据边界必须先于代码边界
- 运行态与长期态必须分开
- 任务内可以中心化，系统级保持去中心化
