# 安全与信任模型

## 1. 目标

本规范定义 Honeycomb 在执行面和进化面中最基础的信任边界与安全约束。

目标不是第一版就做完整安全体系，而是先把后续一定会影响架构的边界定清楚，避免执行、恢复、权限和多租户实现后再整体返工。

## 2. 威胁模型摘要

如果把 Honeycomb 看成一个会执行任务、派生工蜂、读写运行态与长期态的系统，那么当前阶段最值得优先防的威胁是：

- 未授权节点冒充合法工蜂接入蜂后
- 工蜂跨任务、跨命名空间或跨租户越界访问
- 运行时节点直接污染 `skills/` 或 `evolution/` 的长期状态
- 恶意或错误结果绕过校验进入长期默认值
- 恢复流程绕过原本的身份与边界校验

如果是面向部署和共享环境阅读，建议优先阅读：

- 第 4 节：信任边界
- 第 5 节：最小身份与信任材料
- 第 6 节：蜂后与工蜂的信任规则
- 第 7 节：运行目录写入边界
- 第 8 节：多用户与多租户约束

## 3. 核心问题

安全与信任模型需要回答：

- 谁可以启动任务
- 谁可以成为蜂后
- 工蜂如何证明自己属于某个蜂后
- 哪些消息可以被信任
- 哪些文件或目录允许被写入
- 租户、团队、用户之间如何防止越界访问
- 进化面的长期写入如何防止被运行时伪造

## 4. 设计原则

### 4.1 先边界，后强化

第一版可以先采用轻量信任材料和本地部署假设，但边界模型必须先完整。

### 4.2 默认最小信任

任何节点默认都不应被信任到可以跨任务、跨租户、跨命名空间访问。

### 4.3 运行态和长期态分离

运行时节点只能写运行态目录。
长期定义和长期进化记录必须经过显式治理路径。

### 4.4 先校验身份，再校验权限

建议顺序：

1. 身份校验
2. 租户与命名空间边界校验
3. 权限动作校验
4. 资源与配额校验

## 5. 信任边界

建议至少明确以下信任边界：

- 用户与系统边界
- 蜂后与工蜂边界
- 执行面与进化面边界
- 运行态目录与长期定义目录边界
- 租户之间边界
- 任务之间边界

## 6. 最小身份与信任材料

第一版建议至少使用：

- `tenant_id`
- `task_id`
- `namespace`
- `node_id`
- `queen_node_id`
- `protocol_version`
- `queen_token` 或等价临时信任材料

其中：

- `queen_token` 用于证明工蜂确实属于某个蜂后派生的任务蜂群
- 该材料应当是任务级、短期的，不应被设计成长期全局凭证

## 7. 蜂后与工蜂的信任规则

### 7.1 工蜂连接蜂后前必须校验

建议至少校验：

- `task_id` 一致
- `tenant_id` 一致
- `namespace` 一致
- `queen_node_id` 一致
- `queen_token` 有效
- `protocol_version` 可接受

### 7.2 工蜂默认只能信任一个蜂后

第一版建议：

- 一个工蜂只连接一个蜂后
- 一个工蜂只属于一个任务
- 工蜂不得在未经重新初始化的情况下迁移到另一个任务

### 7.3 蜂后不得盲信工蜂结果

蜂后应至少校验：

- 返回的 `assignment_id` 是否存在
- 返回方向是否符合协议
- 结果结构是否符合当前 assignment 的预期
- 工蜂是否仍处于合法任务上下文内

## 8. 运行目录写入边界

建议明确：

- 执行面节点可写 `runtime/`
- 执行面节点不得直接改写 `skills/`
- 执行面节点不得直接改写 `evolution/` 中的长期结论文件
- 进化面可以写 `evolution/`
- 长期稳定定义的正式变更应通过治理流程写入 `skills/`

## 9. 多用户与多租户约束

### 9.1 租户优先隔离

所有权限与信任校验都必须先做租户边界判断。

### 9.2 命名空间是第二层边界

即使同租户，也必须继续校验：

- `global`
- `team/<team_id>`
- `user/<user_id>`

### 9.3 工蜂不得跨租户协作

第一版明确不支持：

- 跨租户任务协作
- 跨租户工蜂连接
- 跨租户自动晋升

## 10. 消息层安全要求

协议消息建议至少满足：

- 明确发送方和接收方
- 带有任务、租户、命名空间上下文
- 有明确的协议版本
- 可被记录进事件、追踪和审计

第一版可以先不做消息签名，但结构上应为以后增加签名、摘要或外部信任材料留空间。

## 11. 进化面的信任要求

进化面处理长期结论，因此信任要求更高。

建议：

- 执行面只能提交评估输入，不直接写长期晋升结果
- 进化面负责生成评分、晋升、淘汰和谱系更新
- 所有长期变更都应附带审计记录
- 晋升应能回指执行证据与评估证据

## 12. 与故障恢复的关系

恢复逻辑必须遵守安全边界。

例如：

- 工蜂重连时不能跳过身份校验
- 任务恢复时不能跨租户拉取事件
- 蜂后崩溃后的手动恢复不能默认为信任所有存活工蜂

## 13. 第一版建议

第一版建议做到：

- 任务级临时信任材料
- 蜂后与工蜂最小身份校验
- 运行态与长期态写入边界清晰
- 租户与命名空间先行校验
- 安全相关拒绝进入审计和追踪

第一版不建议承诺：

- 分布式零信任网络
- 完整证书体系
- 跨机器自动信任发现
- 密钥轮换自动化

## 14. 总结

Honeycomb 的安全与信任模型应围绕三条主线建立：

- 谁是谁
- 谁能连谁
- 谁能写哪里

只要这三条主线清晰，后续不论是本地单机部署还是共享部署，都不会轻易推翻整体结构。
