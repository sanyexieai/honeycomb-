# 资源控制与配额

## 1. 目标

本规范定义 Honeycomb 如何控制：

- 运行中的任务数量
- 蜂后可派生的工蜂数量
- 并行派发数量
- 不同能力的并发上限
- 用户、团队、租户的资源边界

## 2. 分层检查顺序

建议顺序：

1. 租户配额
2. 团队配额
3. 用户配额
4. 任务配额
5. 蜂后工蜂上限
6. 能力并发上限

## 3. 资源维度

建议至少考虑：

- 运行中任务数
- 运行中工蜂数
- 活跃派发数
- 能力并发数
- 预算上限

## 4. 最小配额对象

建议包含：

- 租户配额
- 团队配额
- 用户配额
- 任务配额

任务级建议字段：

- `max_workers`
- `max_parallel_assignments`
- `max_cost`
- `max_runtime_seconds`
- `priority`

## 5. 蜂后规则

- 蜂后不得无限派生工蜂
- 优先复用空闲工蜂
- 不应为无法执行的派发提前派生工蜂

## 6. 工蜂规则

第一版建议：

- 一个工蜂同一时间只处理一个派发
- 一个工蜂只属于一个任务
- 工蜂不跨租户迁移

## 7. 审计与追踪要求

配额拒绝、排队、跳过和限流都应被记录到追踪与审计中。

## 8. 当前建议

第一版先做硬限制和本地计数，不急着做复杂公平调度。
