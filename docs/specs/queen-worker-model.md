# 蜂后与工蜂模型（Bee 运行时版）

## 1. 基本定义

蜂后与工蜂属于 `honeycomb-bee` 运行时，不再由 `honeycomb` 承担主执行循环。

- 蜂后（queen）：任务内规划、派发与汇总的主协调角色
- 工蜂（worker）：执行单个工作单元并回传结果的执行角色

## 2. 核心约束

- queen/worker 执行链路必须 LLM 驱动
- 职责提示词文档必须参与执行并可版本追踪
- 工具调用必须来自能力中心下发的受控工具清单
- bee 运行时不得直接写长期能力注册表与最佳实践

## 3. 蜂后职责

蜂后负责：

- 读取任务拓扑与任务上下文
- 基于 LLM 推理生成派发计划
- 派生或唤醒工蜂并下发 assignment
- 汇总工蜂结果并更新任务内状态
- 在任务结束时发起回收、归档与证据上报

蜂后不负责：

- 跨任务长期能力治理
- 全局最佳实践发布
- 全局默认实现策略写入

## 4. 工蜂职责

工蜂负责：

- 向蜂后注册并声明可执行能力
- 接收单个派发单元
- 使用 LLM + 职责提示词 + 工具完成执行
- 回传进度、结果、工具调用轨迹与证据
- 接受关闭命令并清理会话态

工蜂不负责：

- 理解完整任务图
- 决定长期能力默认值
- 执行跨任务治理动作

## 5. 运行时记忆与上下文

- 任务上下文、会话状态、短期记忆槽位均属于 bee 运行时
- 所有关键决策与工具调用必须可审计与可重放
- 任务内临时结论不可直接提升为全局最佳实践

## 6. 组织原则

- 单任务内允许 queen 主导
- 系统级不设唯一全局中心
- worker 默认单任务单蜂后绑定
- 运行时智能执行与长期能力治理强隔离

## 7. 当前结论

1. queen/worker 是 `honeycomb-bee` 的正式能力，不再是执行面占位角色。
2. 「蜂后理解拓扑并派发」属于默认能力，不再是后续可选增量。
3. bee 只做运行时执行，长期能力资产由 `honeycomb` 与 `honeycomb-evolution` 负责。
