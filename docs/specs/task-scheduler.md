# 任务与调度器规范（Bee Runtime 版）

## 1. 目标

本规范定义任务在 `honeycomb-bee` 中如何被接收、排队、派发、推进与完成，并与能力中心/进化中心形成边界清晰的数据流。

## 2. 并行层级

建议区分三层并行：

### 2.1 任务级并行

多个任务同时运行。

### 2.2 会话/派发级并行

同一任务中多个 assignment 或多个 bee 会话并行。

### 2.3 实现体竞争并行

同一能力的多个实现体并行试跑，仅用于评估，不作为默认业务主路径。

## 3. 调度器职责（bee 内）

调度器负责：

- 接收并排队任务
- 校验租户、命名空间、资源配额
- 驱动 queen 读取拓扑并生成 ready assignment
- 分发 assignment 给 worker 或本地执行单元
- 维护任务状态与中间证据
- 触发失败恢复与重试策略

## 4. 调度器不负责

- 修改长期能力注册表
- 发布全局最佳实践
- 执行跨任务治理决策

以上由 `honeycomb` 与 `honeycomb-evolution` 承担。

## 5. 任务状态

最少状态：

- `queued`
- `running`
- `completed`
- `failed`
- `cancelled`
- `interrupted`

## 6. 调度原则

- 先检查租户与命名空间边界
- 再检查资源与配额
- 再由 queen 基于拓扑与上下文生成 ready assignment
- 优先复用可用 worker 会话
- 所有调度决策必须可追踪、可审计、可重放

## 7. 第一版落地要求（激进口径）

- 支持单机和多进程 bee 运行
- 支持任务级并行与 assignment 级并行
- 支持基于拓扑的 ready 推进
- 支持配额拒绝/允许与失败有限重试
- 支持 LLM 决策与工具调用证据记录

## 8. 与长期治理的数据关系

- bee 调度器产出任务结果、事件、trace、audit、工具调用与评估输入
- evolution 消费全量输入并产出治理决策
- hive 接收治理决策后更新默认实现与实践配置
