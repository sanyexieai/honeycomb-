# 任务拓扑

本文档定义 Honeycomb 中单个任务内部的组织结构，即 `honeycomb-bee` 中 queen 与 worker 如何协作执行。

## 1. 设计目标

任务拓扑要回答：

- 任务内部有哪些工作节点
- 哪些节点可以并行
- 哪些节点有依赖关系
- 蜂后如何派发工作
- 第一版支持哪些拓扑

## 2. 基本原则

### 2.1 任务是容器

任务是工作容器，不等于某一种固定拓扑。

### 2.2 拓扑是任务内部结构

拓扑描述任务内部各个工作节点之间的关系。

### 2.3 蜂后负责解释和推进拓扑

蜂后负责基于 LLM 推理与上下文，把拓扑展开成可以执行的派发单元。

## 3. 推荐的拓扑类型

当前建议统一支持：

- `singleton`
- `pipeline`
- `graph`
- `swarm`

## 4. 各拓扑定义

### 4.1 `singleton`

最简单的任务结构。

特点：

- 只有一个工作节点
- 可以由 queen 执行，也可以委托给一个 worker
- 没有依赖图

### 4.2 `pipeline`

顺序链式结构。

特点：

- 上一步完成后下一步才能开始
- 适合稳定流程
- 调试成本低

### 4.3 `graph`

通用依赖图结构。

特点：

- 某些节点有前置依赖
- 某些节点可并行
- queen 按依赖推进 ready 节点

### 4.4 `swarm`

较松散的蜂群协作结构。

特点：

- 关系不完全固定
- 适合探索性任务
- 协议和恢复复杂度更高

## 5. 第一版支持

第一版正式支持：

- `singleton`
- `pipeline`
- `graph`

`swarm` 暂不实现，但保留结构位与协议扩展点。

## 6. 与蜂后和工蜂的关系

### 6.1 蜂后职责

蜂后负责：

- 读取拓扑
- 基于 LLM + 提示词文档 + 任务上下文计算 ready 节点
- 生成派发单元
- 分配工蜂
- 汇总结果并推进后续节点

### 6.2 工蜂职责

工蜂只负责：

- 接收单个派发单元
- 执行派发单元
- 回报进度与结果

工蜂不理解整个任务图，也不参与长期能力治理。

## 7. 派发单元抽象

建议蜂后不直接把整个任务交给工蜂，而是交付 `assignment`。

建议字段：

- `assignment_id`
- `task_id`
- `node_ref`
- `capability`
- `input`
- `context`
- `constraints`
- `upstream_refs`

## 8. 运行时推进规则

推荐推进流程：

1. 找到依赖已满足的节点
2. queen 基于上下文与策略生成 ready 派发单元
3. 分配给工蜂或本地执行
4. 收集结果
5. 更新图状态
6. 推进下一批节点

## 9. 失败语义

节点失败后，由蜂后决定：

- 重试
- 更换工蜂
- 跳过
- 整体任务失败

失败策略属于任务节点元数据，不应写死在工蜂内部。

## 10. 边界约束

- 拓扑推进与 assignment 生成属于 bee 运行时职责
- 拓扑执行证据必须进入事件、trace、audit
- 拓扑执行中的临时结论不得直接写入全局最佳实践
- 长期拓扑优化建议由 evolution 输出，由 hive 发布
