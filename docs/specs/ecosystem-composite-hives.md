# 生态层组合蜂巢模型

## 1. 目标

本规范定义 Honeycomb 如何支持用户在生态层组合出高层应用蜂巢，而不是在系统内预定义固定职业角色蜂巢。

这里要解决的核心问题是：

- 系统层提供哪些基础积木
- 用户如何利用这些积木组合出高层蜂巢
- 高层蜂巢如何注册、发布、演化和复用
- 如何避免把应用层工作流硬编码进框架本身

## 2. 核心原则

### 2.1 系统不预定义固定职业角色

系统不应内建并硬编码如下对象：

- 规划蜂巢
- 设计蜂巢
- 编码蜂巢
- 审查蜂巢
- 测试蜂巢
- 发布蜂巢
- 运维蜂巢

这些都应被视为：

- 生态层的高层应用蜂巢
- 由用户利用系统能力自行构建出来的结果

### 2.2 系统提供元能力，不提供固定工作流

系统层应提供：

- 蜂巢模型
- 技能模型
- 工具接入
- 任务与常驻运行模型
- 触发器与调度
- 模板与最佳实践
- 注册与发布
- 评分与进化
- 审批与治理

具体工作流如何组织，留给生态层。

## 3. 组合蜂巢的来源

一个高层应用蜂巢通常由以下要素组合而成：

- 基础蜂巢
- skill
- tool
- template
- practice profile
- trigger
- workspace
- governance rule

换句话说，高层蜂巢不是单一文件，而是一个经过组合和约束后的能力体。

## 4. 组合蜂巢的典型结构

建议把高层应用蜂巢理解为：

- 一个面向某类目标的组合定义
- 其中包含多个基础能力引用
- 包含执行策略、触发方式、审批要求和发布边界

例如，一个“代码审查型蜂巢”不应该是系统内建对象，而应是用户在生态中组合出的：

- 若干技能
- 若干模板
- 若干工具
- 一个运行策略
- 一套治理要求

## 5. 系统层与生态层边界

### 5.1 系统层负责什么

系统层负责：

- 定义运行时和长期状态边界
- 提供协议、存储、调度、进化和治理能力
- 提供组合所需的基础模型

### 5.2 生态层负责什么

生态层负责：

- 组合实际业务蜂巢
- 形成领域工作流
- 沉淀行业或团队内高层模板
- 发布可复用的应用型蜂巢

## 6. 注册与发布

组合蜂巢也应该有正式登记方式，但它的层级应高于单个技能。

建议至少记录：

- 组合蜂巢 ID
- 所属租户和命名空间
- 组合内容引用
- 模板与最佳实践引用
- 默认触发方式
- 默认治理边界
- 当前状态与版本

## 7. 与模板的关系

模板是构建组合蜂巢的重要材料，但模板不等于组合蜂巢本身。

可以理解为：

- 模板给出常见结构
- 最佳实践给出局部用法建议
- 组合蜂巢把它们装配成完整能力体

## 8. 与进化的关系

高层组合蜂巢本身也可能演化，但其演化不应绕过基础层。

建议：

- 基础技能先演化
- 模板和最佳实践逐步更新
- 组合蜂巢通过显式版本升级吸收这些变化

不要让组合蜂巢隐式漂移，否则会失去可追踪性。

## 9. 与审批的关系

越高层的组合蜂巢，越可能影响更大范围的行为。

因此建议：

- 用户级组合蜂巢可先本地试验
- 团队级组合蜂巢应经过更明确的审查和发布
- 全局组合蜂巢必须具备更强治理要求

## 10. 第一版建议

第一版不需要真的实现完整生态市场，但应预留：

- 组合蜂巢的概念
- 组合蜂巢的注册字段
- 组合蜂巢的版本与发布路径
- 系统层和生态层的边界说明

## 11. 总结

Honeycomb 的系统层应提供“构建工作流蜂巢的能力”，而不是直接内建一套固定职业角色。

真正的高层能力应来自生态层，由用户利用蜂巢、技能、工具、模板、最佳实践和治理能力组合生成。
