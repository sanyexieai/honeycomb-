# 蜂运行时、蜂巢能力与生态进化（三层激进重构）

## 1. 目标

本文档定义 Honeycomb 下一阶段的硬边界，不保留旧路径兼容：

- 新增独立 `honeycomb-bee` 二进制，承载蜂后/工蜂运行时
- 蜂执行核心强制采用 LLM 驱动
- 蜂巢能力中心与生态进化中心分别承担长期能力与全局治理
- 拒绝“单一二进制持续膨胀”的实现路线

## 2. 三层模型

### 2.1 蜂运行时层（Bee Runtime）

由 `honeycomb-bee` 承担，职责是任务内实时执行：

- 以 `queen` 或 `worker` 角色运行
- 使用 LLM + 职责提示词文档 + 工具集合执行工作单元
- 维护任务内会话、短期上下文、局部状态与产物
- 与控制/能力中心通过正式协议交互

硬约束：

- 蜂执行路径必须可追溯到 LLM 实现体，不允许隐藏式硬编码工作流替代
- 蜂不直接写长期注册表、最佳实践、全局默认策略

### 2.2 蜂巢能力层（Hive Capability Center）

由 `honeycomb` 承担，职责是长期能力资产管理：

- skill、tool、implementation、template、practice 的登记与发布
- 全局工具白名单、能力约束、默认实现策略
- 为蜂运行时提供可解析能力清单与版本化引用

定位：蜂巢层回答“系统长期拥有哪些可用能力，以及默认怎么用”。

### 2.3 生态进化层（Ecology Evolution Center）

由 `honeycomb-evolution` 承担，职责是跨任务治理：

- 全量运行观测汇总、质量评分、谱系维护
- 晋升/淘汰与最佳实践沉淀
- 全局能力优化建议与治理执行

定位：生态层回答“未来应该保留什么、淘汰什么、推广什么”。

## 3. 二进制边界（强制）

主二进制固定为 3 个：

1. `honeycomb-bee`（运行时执行）
2. `honeycomb`（能力中心）
3. `honeycomb-evolution`（生态进化）

不再采用“把新增能力继续堆进 `honeycomb` 执行入口”的路线。

## 4. LLM 驱动原则（强制）

### 4.1 核心定义

“核心蜂由 LLM 驱动”指：

- queen/worker 在执行工作单元时，决策与生成过程由 LLM 能力链完成
- 职责提示词文档是正式执行输入，而不是注释性资料
- 工具调用通过受控能力清单接入，不允许绕过工具治理

### 4.2 非目标

以下路径不再视为目标能力：

- 仅靠静态规则引擎模拟“智能调度”并替代 LLM 决策
- 将提示词固化为不可追踪代码常量且缺乏版本治理
- 在运行时直接引入长期治理写入逻辑

## 5. 记忆与上下文分层

### 5.1 运行时记忆与上下文（Bee）

- 任务上下文、会话状态、短期记忆槽位、中间产物
- 必须可重放、可审计、可清理
- 生命周期随任务或会话结束而衰减或归档

### 5.2 长期记忆与最佳实现（Hive + Evolution）

- 全局最佳实践、实现体谱系、推荐策略、长期知识沉淀
- 由能力中心登记，由进化中心评估与治理
- 不允许由 bee 运行时直接落长期写入

## 6. 最小交互协议（方向）

- bee 向能力中心请求能力定义与可用工具清单
- bee 回传执行证据、产物引用、工具调用轨迹
- 进化中心消费全量证据并产出治理决策
- 能力中心按治理决策发布新默认实现或实践

## 7. 落地原则

- 不做兼容别名，不保留旧语义入口
- 模糊边界优先删除，不做“临时折中”
- 任一功能都必须归属到 bee / hive / ecology 三层之一

## 8. 当前结论

- `honeycomb-bee` 是新增且必需的运行时二进制
- `honeycomb` 不再承担蜂运行时执行主入口，转为能力中心
- `honeycomb-evolution` 负责全量监控视角下的能力进化与治理闭环
