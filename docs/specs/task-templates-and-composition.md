# 任务模板与编排模板

## 1. 目标

本规范定义 Honeycomb 如何把常见任务结构、拓扑结构和技能组合沉淀成可复用模板。

## 2. 为什么需要模板

如果每次任务都从零拼装，会有几个问题：

- 常见编排重复劳动
- 成熟拓扑无法沉淀
- 技能组合经验难以复用
- 不同用户对同类任务的组织方式差异过大

模板层的作用是把“重复出现且验证过有效的结构”固化下来。

## 3. 模板分类

建议至少区分：

- 任务模板
- 拓扑模板
- 技能组合模板

### 3.1 任务模板

描述某类任务常见的：

- 输入结构
- 目标结构
- 约束
- 默认拓扑

### 3.2 拓扑模板

描述某类常见图结构，例如：

- 单步
- 顺序流水线
- 汇聚图
- 多分支分析再汇总

### 3.3 技能组合模板

描述某类任务通常需要哪些技能共同工作。

## 4. 模板内容建议

建议模板至少包含：

- `template_id`
- `name`
- `tenant_id`
- `namespace`
- `template_type`
- `description`
- `default_topology`
- `default_skill_refs`
- `constraints`
- `version`
- `status`

## 5. 与任务的关系

模板不是任务实例。

模板负责给出默认结构，真正的任务在运行时仍会生成独立的：

- `TaskSpec`
- `TaskRuntime`
- `TaskHiveSession`

## 6. 与最佳实践的关系

模板和最佳实践不是同一层：

- 模板关注任务和编排结构
- 最佳实践关注某个技能在某类上下文下的推荐用法

但它们会互相影响：

- 模板可引用最佳实践
- 最佳实践积累多了也可能推动模板更新

## 7. 推荐使用方式

建议新任务启动时：

1. 先匹配任务模板
2. 再展开默认拓扑
3. 再根据上下文匹配技能最佳实践
4. 生成本次任务的实际执行计划

## 8. 第一版建议

第一版建议先支持：

- 本地静态模板文件
- 任务模板与拓扑模板
- 模板到 `TaskSpec` 的展开
- 模板状态管理与版本记录

第一版不必承诺：

- 自动模板生成
- 模板市场推荐
- 多模板自动融合

## 9. 总结

模板层的意义，是把“反复出现的有效组织方式”从临时经验提升为正式资源。
