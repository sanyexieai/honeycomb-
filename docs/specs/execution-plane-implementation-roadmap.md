# Bee 运行时最小实现路线图

## 1. 目标

本规范把当前已经明确的 `honeycomb-bee` 运行时设计，收敛成一条可实际落地的最小实现路线。

目标不是一次实现所有能力，而是在不破坏三层架构的前提下，分阶段建立可运行、可扩展、可重构的 bee 运行时基础。

## 2. 实现原则

bee 实现建议遵守：

- 先跑通最小闭环，再扩展复杂能力
- 先把边界和数据模型立稳，再补优化
- 先支持逻辑复制，再预留物理复制
- 先文件系统落地，再考虑更复杂基础设施
- 先单机可验证，再考虑多节点或更强分布式能力

## 3. 第一阶段目标

第一阶段只要求做到：

- `honeycomb-bee` 支持不同角色启动
- 蜂后能启动一个工蜂
- 工蜂能连接回蜂后
- 蜂后能派发一个最小 assignment
- 工蜂能返回结果
- 任务、节点、事件、追踪能落盘

这一步是 bee 运行时骨架验证。

## 4. 推荐阶段划分

### 阶段 1：基础运行骨架

建议实现：

- `--role queen|worker`
- 节点身份对象
- 最小任务对象
- 最小任务目录写入
- 最小事件记录

交付标准：

- 能启动 queen
- 能启动 worker
- 能写出 task 与 node 基本状态

### 阶段 2：最小协议闭环

建议实现：

- 本地通信通道
- `hello`
- `hello_ack`
- `heartbeat`
- `shutdown`

交付标准：

- worker 能注册成功
- queen 能维护 worker 生命周期

### 阶段 3：最小派发闭环

建议实现：

- assignment 对象
- `task_assign`
- `task_progress`
- `task_result`
- assignment 落盘

交付标准：

- queen 能把一个 assignment 下发给 worker
- worker 能返回结果
- queen 能记录完成状态

### 阶段 4：任务运行时闭环

建议实现：

- `TaskSpec`
- `TaskRuntime`
- `TaskHiveSession`
- `events.jsonl`
- `trace.jsonl`
- `audit.jsonl`

交付标准：

- 一个任务执行完后，状态、事件、追踪、审计完整可查

### 阶段 5：任务拓扑推进

建议实现：

- `singleton`
- `pipeline`
- `graph`
- ready 节点推进
- assignment 依赖处理

交付标准：

- 同一任务内可执行多节点拓扑

### 阶段 6：常驻蜂巢与触发器

建议实现：

- 常驻蜂巢启动模型
- 手动触发
- 定时触发
- 事件触发
- 触发状态与历史记录

交付标准：

- 可以让一个常驻蜂巢持续运行并按触发器工作

### 阶段 7：资源与恢复基础

建议实现：

- 任务级 `max_workers`
- 基本配额检查
- heartbeat 超时检测
- assignment 超时
- bounded retry

交付标准：

- bee 运行时在失败和负载下仍有最小稳定性

### 阶段 8：能力中心读取对接

建议实现：

- bee 读取能力中心中的正式技能状态
- 读取默认实现体
- 读取模板和最佳实践引用

交付标准：

- bee 不再依赖手写目录假设，而是依赖能力中心正式发布结果

## 5. 推荐模块顺序

如果按 Rust 模块来实现，建议顺序如下：

1. 核心类型层
2. 文件存储层
3. 节点身份层
4. 协议层
5. 任务运行时层
6. 调度推进层
7. 常驻蜂巢与触发器层
8. 资源控制与恢复层
9. 注册表读取层

## 6. 推荐最小代码边界（bee）

建议至少先明确这些模块边界：

- `core`：核心对象与枚举
- `protocol`：蜂后工蜂通信协议
- `runtime`：任务、节点、会话、状态
- `storage`：文件系统读写
- `scheduler`：任务拓扑推进
- `resident`：常驻蜂巢与触发器
- `control`：配额与策略读取
- `registry_client`：能力中心读取客户端

## 7. 第一版可以暂时不做的内容

bee 第一版可以明确暂缓：

- 工蜂物理复制真正启用
- 图形化控制台
- 复杂公平调度
- 跨机器通信
- 自动蜂后接管
- 远程仓库分发

## 8. 验证顺序建议

每阶段都应有可验证目标：

1. 启动 queen 和 worker
2. 完成最小握手
3. 完成单 assignment 执行
4. 完成单任务落盘
5. 完成多节点拓扑
6. 完成一个常驻蜂巢触发流程
7. 完成一次超时重试
8. 完成从注册表读取技能并执行

## 9. 与代码重构的关系

当前已有原型代码可以作为参考，但不应绑定实现路线。

建议：

- 保留已有类型和示例中有价值的部分
- 不被现有 CLI 结构和目录结构限制
- 以文档中的边界为准，允许较大幅度重构

## 10. 总结

bee 运行时最小实现路线图应该围绕一个原则：

先把“蜂后、工蜂、任务、事件、状态、派发”这条骨架跑通，再逐层把触发器、常驻蜂巢、配额、恢复和能力中心接进来。

这样做，既能保持进展，也不会在一开始把系统写死。
