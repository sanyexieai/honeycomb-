# 调试与查看工具形态

## 1. 目标

本规范定义 Honeycomb 后续应提供哪些最小调试与查看工具，以支撑开发、排错和治理。

## 2. 为什么需要单独设计

当前我们已经有：

- 事件
- 追踪
- 审计
- 任务状态
- 节点状态
- 评估和晋升记录

如果没有清晰的查看工具模型，这些数据虽然存在，但难以真正被使用。

## 3. 最小工具集合建议

建议至少规划：

- 任务查看
- 节点查看
- 事件查看
- 追踪查看
- 审计查看
- 注册表查看
- 进化结果查看

## 4. 任务查看

建议支持：

- 查看任务基本信息
- 查看任务当前状态
- 查看蜂后与工蜂列表
- 查看 assignment 列表
- 查看最近事件和最近错误
- 查看任务产物

## 5. 节点查看

建议支持：

- 查看节点身份
- 查看节点所属任务
- 查看最近心跳
- 查看当前状态
- 查看最近执行记录

## 6. 事件与追踪查看

建议支持：

- 按任务筛选事件
- 按节点筛选事件
- 按 assignment 查看相关追踪链
- 查看失败前后的关键事件

## 7. 审计查看

建议支持：

- 查看谁启动了任务
- 查看谁发布了技能
- 查看谁做了晋升或废弃决策
- 查看拒绝、回滚、恢复等关键动作

## 8. 注册表与进化结果查看

建议支持：

- 查看技能当前版本
- 查看默认实现体
- 查看晋升历史
- 查看候选与已废弃对象
- 查看最佳实践适用上下文

## 9. 工具形式建议

建议分阶段：

### 9.1 第一阶段

- CLI 查看命令
- 基于文件系统读取
- 提供纯文本和 JSON 两种输出

### 9.2 第二阶段

- 本地 Web 查看面板
- 任务图和追踪图可视化
- 工蜂生命周期可视化

## 10. 第一版建议命令

建议至少保留结构位：

- `inspect task <task_id>`
- `inspect node <node_id>`
- `inspect events <task_id>`
- `inspect trace <task_id>`
- `inspect audit <task_id>`
- `inspect skill <skill_id>`
- `inspect evolution <hive_id>`

## 11. 总结

调试与查看工具不是附属功能，而是把现有事件、追踪、审计和治理数据真正变成可用能力的入口。
