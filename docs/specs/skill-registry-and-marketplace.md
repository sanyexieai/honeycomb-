# 技能注册与发布

## 1. 目标

本规范定义 Honeycomb 如何把 `skills/` 从静态目录提升为可管理的系统资源。

这里的重点不是先做公开市场，而是先定义：

- 技能如何被登记
- 技能如何被发现
- 技能如何被安装
- 技能如何被升级
- 技能如何被发布到更大范围
- 技能如何被废弃或回滚

## 2. 为什么需要这一层

如果没有注册与发布机制，`skills/` 很快会出现几个问题：

- 同名技能来源不清
- 技能版本不清
- 哪些技能可被执行不清
- 哪些技能只是实验分支不清
- 哪些技能已经被正式晋升不清

因此，技能注册层本质上是技能治理层，而不是简单的文件列表。

## 3. 基本定义

### 3.1 技能注册表

技能注册表用于记录系统已知技能的元信息。

建议记录：

- 技能 ID
- 名称
- 所属租户
- 所属命名空间
- 当前版本
- 当前状态
- 可见性
- 来源
- 安装位置
- 默认实现体

### 3.2 技能包

技能包表示一组可安装、可迁移的技能定义与资源集合。

它至少应包含：

- 技能定义文件
- 实现体定义
- 基因定义
- 最佳实践文件
- 相关资源文件
- 版本与元数据

### 3.3 发布

发布表示把某个技能从一个作用域晋升或复制到更大范围。

例如：

- 从 `user/<user_id>` 发布到 `team/<team_id>`
- 从 `team/<team_id>` 发布到租户内 `global`

## 4. 技能生命周期

建议技能至少有这些状态：

- `draft`：草稿
- `registered`：已登记
- `active`：可执行
- `deprecated`：已废弃
- `archived`：归档

### 4.1 草稿

仅表示技能资源存在，但尚未进入正式注册表。

### 4.2 已登记

表示技能已经被系统识别，可被查看和验证。

### 4.3 可执行

表示技能通过最小校验，可被任务选择和执行。

### 4.4 已废弃

表示技能不建议继续作为默认选择，但历史记录仍可保留。

### 4.5 归档

表示技能退出活跃范围，仅供历史追溯和回放使用。

## 5. 注册流程建议

建议最小流程：

1. 技能目录进入某个命名空间路径
2. 系统扫描并读取元信息
3. 执行结构校验
4. 写入技能注册表
5. 标记状态为 `registered` 或 `active`

第一版可以先由 CLI 或后台扫描完成，不需要一开始就做复杂服务化。

## 6. 安装流程建议

安装技能时建议区分三种来源：

- 本地技能目录
- 导入的技能包
- 从更高范围复制或发布下来的技能

安装后应至少做：

- 结构校验
- 路径登记
- 版本记录
- 所属作用域记录

## 7. 版本模型

建议每个技能至少有：

- 技能版本
- 实现体版本
- 协议版本
- 文档或元数据版本

关键原则：

- 技能版本不等于实现体版本
- 发布行为应明确版本来源
- 回滚时应以注册表记录为准，而不是只看目录内容

## 8. 命名空间与发布规则

技能注册和发布必须遵守：

- 先租户边界
- 再命名空间边界
- 再可见性与权限判断

默认发布路径建议：

- `user/<user_id>` -> `team/<team_id>`
- `team/<team_id>` -> `global`

第一版不建议支持：

- 跨租户直接发布
- 未经审计的直接全局覆盖

## 9. 注册表建议字段

建议每条注册记录至少包含：

- `skill_id`
- `name`
- `tenant_id`
- `namespace`
- `visibility`
- `status`
- `version`
- `default_implementation_id`
- `installed_path`
- `source_type`
- `source_ref`
- `created_at`
- `updated_at`

## 10. 存储建议

建议新增独立目录：

```text
registry/
  tenant/
    <tenant_id>/
      skills/
        index.json
        user/
        team/
        global/
```

或者：

```text
control/
  tenant/
    <tenant_id>/
      registry/
        skills.json
```

重点是：注册信息不应只依赖目录扫描结果，而应有正式记录。

## 11. 与进化面的关系

进化面不直接等于技能注册层，但二者会交互：

- 进化面可以提出新的默认实现体
- 进化面可以提出新的最佳实践发布建议
- 正式写入技能注册状态仍应走注册或治理路径

也就是说：

- 进化面产出建议
- 注册与治理层决定如何落到稳定技能资源上

## 12. 与权限模型的关系

技能注册与发布至少涉及这些动作：

- `read`
- `execute`
- `write_definition`
- `promote`
- `deprecate`

发布和废弃必须留下审计记录。

## 13. 与运行时的关系

运行时只应读取技能注册结果，不应负责管理注册表。

建议：

- bee 运行时经能力中心读取“当前可执行技能集合”
- 任务选择技能时依据注册状态与权限边界
- 运行时不得直接把临时试验写成正式技能注册变更

## 14. 第一版建议

第一版建议做到：

- 有正式技能注册表
- 有最小技能状态模型
- 支持本地登记与扫描
- 支持用户、团队、全局三级发布
- 支持废弃标记
- 支持回指安装路径和来源

第一版不必承诺：

- 公开在线市场
- 外部下载中心
- 复杂评分排序推荐
- 自动依赖求解

## 15. 总结

技能注册与发布层的核心作用，是让 `skills/` 从“目录集合”变成“可治理资源集合”。

这样后续无论是做权限、进化、发布、回滚还是团队共享，都不会再依赖隐式约定。
