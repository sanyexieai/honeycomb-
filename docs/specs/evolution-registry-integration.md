# 进化面与注册表联动

## 1. 目标

本规范定义 Honeycomb 的进化面如何把长期评估和治理结果，安全地映射到技能注册表与稳定技能资源上。

这份文档要解决的核心问题是：

- 评分结果什么时候只是参考
- 什么时候可以形成治理决策
- 治理决策如何影响注册表
- 注册表如何影响后续执行默认值
- 哪些内容能进入稳定技能定义，哪些只能停留在进化记录中

## 2. 为什么要单独设计这一层

如果没有“进化面与注册表联动”这层，系统会出现两个常见问题：

- 进化面有很多评分和建议，但不会真正改变系统默认行为
- 运行时或评估结果会绕过治理，直接污染长期定义

所以这层的职责不是“再做一次评估”，而是定义：

- 长期建议如何落地
- 默认值如何更新
- 什么能被发布，什么不能

## 3. 基本原则

### 3.1 进化面产出建议，注册表承接正式状态

建议分工：

- 进化面负责生成评估、建议、晋升决策、淘汰决策
- 注册表负责保存系统正式可见的技能状态与默认指向

### 3.2 执行面不能直接改注册表默认值

执行面只能产出：

- 任务结果
- 指标
- 评估输入
- 运行证据

执行面不得直接把某次运行结果写成：

- 默认实现体
- 正式最佳实践
- 注册表当前状态

### 3.3 注册表更新必须可审计、可回滚

任何由进化面推动的注册表变更，都必须：

- 有明确来源
- 有时间戳
- 有决策记录
- 可回退到之前状态

## 4. 联动对象

进化面与注册表联动时，建议至少涉及：

- `skill`
- `implementation`
- `practice profile`
- `default_implementation_id`
- `status`
- `visibility`
- `version`

## 5. 典型联动场景

### 5.1 更新默认实现体

当某个实现体经过持续评估后表现更好，进化面可以提出：

- 把 `default_implementation_id` 从 `impl_v1` 改为 `impl_v2`

这时不应直接覆盖，而应：

1. 形成治理决策
2. 更新注册表记录
3. 写入晋升审计
4. 保留旧默认值历史

### 5.2 晋升最佳实践

当某个最佳实践在特定上下文下稳定表现更好时，进化面可以提出：

- 将其从候选状态变成正式最佳实践

注册表或与之配套的索引层应能指向：

- 当前哪些最佳实践是正式有效的
- 哪些仍是候选或观察中

### 5.3 废弃实现体

当某个实现体长期表现差、存在兼容问题或安全风险时，进化面可以提出：

- 将其标记为 `deprecated`

注册表更新后，执行面默认不应再优先选择它。

### 5.4 发布到更大范围

当用户级对象被晋升到团队级或全局范围时，进化面可以发出发布建议。

真正的范围变更仍建议通过注册表和发布流程完成，而不是只改进化记录。

## 6. 推荐状态流转

建议至少区分以下状态：

- `candidate`
- `active`
- `deprecated`
- `archived`

联动时可参考：

- 进化面把 `candidate -> recommend_active`
- 注册表经过治理后把该对象标记为 `active`
- 老默认值可被标记为 `deprecated` 或继续保留为非默认对象

## 7. 推荐数据流

建议采用以下链路：

1. 执行面产出运行结果与评估输入
2. 进化面生成 `FitnessReport`
3. 进化面生成治理决策记录
4. 注册层消费治理决策记录
5. 注册表更新正式状态
6. 执行面下一次读取新的正式状态

这里最关键的是：

- 执行面读注册表
- 进化面写治理建议
- 注册层写正式状态

## 8. 联动记录建议

建议新增一种显式记录：

- `RegistryUpdateRecord`

建议字段：

- `update_id`
- `tenant_id`
- `namespace`
- `skill_id`
- `target_type`
- `target_id`
- `action`
- `old_value`
- `new_value`
- `source_decision_id`
- `created_at`

这样后面可以明确看到：

- 注册表为什么变了
- 是谁推动变更的
- 依据了哪次治理决策

## 9. 存储建议

建议增加：

```text
registry/
  tenant/
    <tenant_id>/
      skills/
        updates/
          update_001.json
```

或者放在：

```text
control/
  tenant/
    <tenant_id>/
      registry/
        updates/
```

关键不是具体目录名，而是要有单独的“注册表更新记录层”。

## 10. 与回滚的关系

注册表联动必须天然支持回滚。

建议：

- 每次正式变更都保留旧值
- 回滚时通过历史 `RegistryUpdateRecord` 或历史技能包恢复
- 回滚也应写入新的审计和更新记录

## 11. 与权限模型的关系

联动涉及的关键动作：

- `promote`
- `deprecate`
- `write_definition`
- `inspect`

只有通过治理路径获得授权的主体，才可推动注册表正式更新。

## 12. 第一版建议

第一版建议做到：

- 进化面能生成明确治理决策
- 注册层能消费治理决策并更新默认值
- 默认实现体更新有历史记录
- 最佳实践晋升有正式状态记录
- 所有联动变更可审计、可回滚

第一版不必承诺：

- 全自动无人值守全局晋升
- 复杂审批流引擎
- 多阶段人工签核系统

## 13. 总结

进化面与注册表联动的本质，是把“评分上的更优”转换成“系统中的正式默认”。

这一步必须经过显式、可追溯、可回滚的桥接层，否则 Honeycomb 的长期状态一定会混乱。
