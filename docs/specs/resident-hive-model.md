# 常驻蜂巢模型

## 1. 目标

本规范定义 Honeycomb 中除任务型蜂巢之外的另一类重要对象：常驻蜂巢。

它们用于承担持续运行、监听、整理、服务和定期动作，而不是只为某次短任务存在。

## 2. 为什么需要常驻蜂巢

如果系统目标不只是“执行一次任务”，而是逐步承接：

- UI 状态整理
- 定时提醒
- 健康巡检
- 持续索引
- 长期监控
- 自动准备开发上下文

那么就必须有一类不以单次任务为中心的蜂巢。

## 3. 两类蜂巢

### 3.1 任务型蜂巢

特点：

- 围绕单次任务存在
- 生命周期通常较短
- 目标明确、边界清晰

### 3.2 常驻蜂巢

特点：

- 长时间运行
- 基于触发器、事件、定时或订阅工作
- 持续维护某类系统能力

## 4. 常驻蜂巢的典型用途

常驻蜂巢不限定于 UI，也可以包括：

- 提醒蜂巢
- 巡检蜂巢
- 视图状态整理蜂巢
- 注册表索引蜂巢
- 任务预热蜂巢
- 事件汇总蜂巢

## 5. 生命周期建议

建议至少包含：

- `configured`
- `starting`
- `running`
- `degraded`
- `paused`
- `stopping`
- `stopped`
- `failed`

## 6. 与任务的关系

常驻蜂巢可以：

- 直接处理轻量长期职责
- 触发新的任务
- 维护某个持续 view model
- 根据规则生成提醒或建议

但它不应默认等于“一个永不结束的普通任务”。

## 7. 输入来源

常驻蜂巢建议支持：

- 定时触发
- 文件或目录变化
- 运行事件订阅
- 审计与追踪流
- 注册表变化
- 外部控制输入

## 8. 输出形式

常驻蜂巢可输出：

- 状态文件
- view model
- 通知事件
- 新任务触发请求
- 索引或摘要结果

## 9. 当前建议

系统层应支持常驻蜂巢这一类基础对象，但不预定义具体用途。

具体要做 UI、提醒、巡检还是别的长期职责，应交给生态层组合出来。
