# 技能包格式与导入导出

## 1. 目标

本规范定义 Honeycomb 中技能如何被打包、导入、导出、迁移和回滚。

目标不是一开始就做复杂包管理器，而是先把技能资源如何从一个环境稳定移动到另一个环境讲清楚。

## 2. 为什么需要技能包

如果没有统一的技能包格式，后面会出现几个问题：

- 技能只能靠目录拷贝迁移
- 技能来源和版本不清楚
- 发布到团队或全局时难以保证一致性
- 回滚时难以还原到明确状态
- 审计时难以回答“这个技能当时是从哪里来的”

技能包层的作用就是把“一个技能目录”变成“一个可迁移、可登记、可审计的资源单元”。

## 3. 基本定义

### 3.1 技能包

技能包是一个可独立导入导出的技能资源集合。

它至少应包含：

- 技能定义文件
- 实现体定义
- 基因定义
- 最佳实践文件
- 元数据文件
- 资源文件

### 3.2 包元数据

包元数据描述这个包是什么、来自哪里、适用于什么范围。

### 3.3 导入

导入表示把一个技能包安装到某个租户或命名空间中，并登记到注册表。

### 3.4 导出

导出表示把当前某个技能及其必要资源打成一个可迁移包。

## 4. 包结构建议

建议采用目录型包结构：

```text
skill-package/
  manifest.json
  skill/
    hive.md
    implementation.json
    genome.json
    practices/
    resources/
```

如果需要单文件交付，可在此目录结构之上再做压缩封装。

## 5. 包元数据建议

`manifest.json` 建议至少包含：

- `package_id`
- `skill_id`
- `name`
- `version`
- `tenant_scope`
- `namespace_scope`
- `source_type`
- `source_ref`
- `created_at`
- `binary_compatibility`
- `protocol_compatibility`
- `content_hash`

## 6. 导出规则

导出时建议：

- 明确导出的技能 ID 和版本
- 记录导出来源作用域
- 明确是否包含最佳实践
- 明确是否包含候选实现体
- 对资源文件生成摘要或哈希

第一版建议导出稳定资源，不导出任务运行态。

## 7. 导入规则

导入时建议至少做：

1. 解析包元数据
2. 校验结构完整性
3. 校验版本兼容性
4. 校验目标租户和命名空间是否允许导入
5. 把内容写入目标技能目录
6. 写入技能注册表
7. 生成导入审计记录

## 8. 导入冲突处理

导入时常见冲突包括：

- 同一 `skill_id` 已存在
- 版本号冲突
- 默认实现体不同
- 最佳实践命名冲突
- 目标作用域不允许覆盖

建议至少支持三种策略：

- `reject`：拒绝导入
- `install_as_new_version`：作为新版本导入
- `install_as_variant`：作为同技能下的新变体导入

第一版不建议默认静默覆盖。

## 9. 与注册表的关系

技能包只是迁移载体，真正的系统可执行状态仍以注册表为准。

也就是说：

- 包导入后，注册表新增或更新一条技能记录
- 包导出前，可以从注册表确定当前正式版本
- 回滚时应优先参考注册表历史和导入记录

## 10. 与进化面的关系

进化面可能产生：

- 新的默认实现体建议
- 新的最佳实践建议
- 新的候选实现体

但这些内容是否进入技能包，应遵守明确策略。

建议第一版：

- 默认只打包稳定定义和已正式发布内容
- 候选内容可选打包，不默认打包
- 运行态证据不放入技能包，而保留在 `runtime/` 和 `evolution/` 中

## 11. 与权限模型的关系

技能包导入导出至少涉及：

- `read`
- `write_definition`
- `promote`
- `inspect`

导入和导出都应留下审计记录。

## 12. 与回滚的关系

技能包是回滚的重要基础。

建议：

- 每次正式发布都能生成对应包或等价快照
- 包元数据中应有版本与来源
- 回滚时能指向明确的历史包或历史登记记录

## 13. 第一版建议

第一版建议做到：

- 目录型技能包
- 有 `manifest.json`
- 支持导入到指定租户和命名空间
- 支持导出当前稳定技能
- 支持冲突拒绝或新版本导入
- 导入导出进入审计

第一版不必承诺：

- 复杂依赖求解
- 远程仓库下载
- 增量包
- 自动合并包内容

## 14. 总结

技能包格式层的核心意义，是让技能资源具备：

- 可迁移性
- 可登记性
- 可审计性
- 可回滚性

这样 `skills/`、注册表和进化面之间的关系才会真正稳定。
