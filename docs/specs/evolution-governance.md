# 进化面治理流程

## 1. 目标

本规范定义 Honeycomb 进化面如何把执行结果、评分结果和治理决策连接起来，形成长期稳定的能力优化流程。

## 2. 为什么需要治理

如果只有评分，没有治理流程，就会出现：

- 某次局部最优被误当成长期全局最优
- 候选实现体没有明确晋升条件
- 被淘汰或废弃的对象没有清晰原因
- 最佳实践和实现体的状态无法追踪

因此，进化面不仅是评分系统，也是治理系统。

## 3. 基本阶段

建议把进化面治理拆成五个阶段：

1. 收集
2. 评估
3. 审核
4. 决策
5. 发布

## 4. 各阶段说明

### 4.1 收集

收集 bee 运行时产出的：

- 结果
- 指标
- 事件
- 追踪
- 审计
- 评估输入

### 4.2 评估

评估阶段生成：

- 评分报告
- 候选比较结果
- 风险提示
- 上下文适用性判断

### 4.3 审核

审核阶段用于判断：

- 这次结果是否足够稳定
- 是否存在明显异常样本
- 是否允许进入更高作用域
- 是否需要人工确认

### 4.4 决策

决策阶段可以产生：

- 晋升
- 保持候选
- 废弃
- 继续观察

### 4.5 发布

发布阶段将治理结果正式写入：

- 注册表
- 最佳实践档案
- 谱系记录
- 审计记录

## 5. 治理对象

建议治理至少覆盖：

- 实现体
- 最佳实践档案
- 默认实现体指向
- 废弃状态
- 谱系状态

## 6. 决策结果类型

建议最少支持：

- `promote`
- `deprecate`
- `hold`
- `observe`
- `rollback`

## 7. 发布边界

进化面治理必须遵守：

- 先租户边界
- 再命名空间边界
- 再权限与可见性判断

默认建议路径：

- `user/<user_id>` 内先形成候选
- 表现稳定后发布到 `team/<team_id>`
- 团队内再验证后发布到 `global`

## 8. 审计要求

每次治理决策都应记录：

- 决策对象
- 决策动作
- 决策依据
- 作用域变化
- 时间
- 决策来源

## 9. 与 bee 运行时的边界

bee 运行时只负责产出事实和评估输入。

进化面负责：

- 做长期判断
- 修改长期状态
- 触发晋升与废弃

bee 运行时不应直接把运行结果写成长期默认值；长期默认由能力中心在治理路径下发布。

## 10. 第一版建议

第一版建议先做：

- 离线或批量治理
- 明确候选与晋升状态
- 用户到团队、团队到全局的显式发布
- 所有决策有审计记录

第一版不必承诺：

- 全自动全局晋升
- 实时在线治理闭环
- 复杂审批工作流引擎

## 11. 总结

进化面治理的本质，不是“让系统自动变强”，而是“让系统以可追踪、可解释、可回滚的方式长期变强”。
