# 存储 Schema 版本、迁移与恢复策略

## 1. 目标

本文档补充 Honeycomb 文件存储层的 schema version、兼容性、迁移和损坏恢复原则。

## 2. 为什么需要单独定义

当前设计明确以文件系统为第一版基础设施。如果没有统一版本与迁移规则，后续一旦：

- 增加字段
- 调整状态枚举
- 拆分目录
- 引入新索引

就会直接影响恢复、回放和进化面读取。

## 3. 版本字段建议

建议以下对象显式携带版本：

- `task.json`
- assignment 记录
- `events.jsonl`
- `trace.jsonl`
- `audit.jsonl`
- `trigger` 记录
- `resident hive` 状态
- 进化面评估输入

建议字段：

- `schema_version`
- `written_at`
- `writer_version`

## 4. 兼容性原则

建议采用：

- 向后兼容读取优先
- 向前兼容尽量忽略未知字段
- 破坏性变化必须提升主版本

版本解释建议：

- `1.x`：可通过默认值或忽略未知字段兼容
- `2.0`：存在结构性破坏，需要迁移

## 5. 迁移类型

建议区分：

- `in_place`：原地补字段或重写小文件
- `copy_forward`：写新文件并保留旧文件
- `rebuild_index`：基于事件流重建索引或快照
- `manual_blocked`：自动迁移风险过高，等待人工处理

## 6. 运行时迁移原则

第一版建议：

- 任务运行中不做破坏性迁移
- 运行中发现旧版本数据时，只允许做兼容读取或只追加旁路文件
- 真正的结构迁移应在任务结束后或离线维护阶段完成

## 7. 事件日志版本策略

`events.jsonl`、`trace.jsonl`、`audit.jsonl` 建议遵守：

- 一旦写入，不做原地重写
- 新字段通过扩展 payload 引入
- 若事件类型语义变化明显，应新增 `event_type`，不要复用旧语义

## 8. 快照文件策略

`task.json` 和其他快照建议：

- 允许原子替换写入
- 写入前生成临时文件
- 成功后再 rename 覆盖
- 保留最近一个可恢复备份，如 `task.json.bak`

## 9. 损坏恢复建议

建议恢复顺序：

1. 读取快照
2. 校验 `schema_version`
3. 若快照损坏，尝试 `.bak`
4. 若仍失败，回退到事件流重建
5. 若事件流也不完整，标记 `manual_intervention_required`

## 10. 索引重建原则

以下内容建议视为可重建索引，而非唯一事实源：

- 当前任务摘要
- 最近错误列表
- assignment 聚合视图
- worker 当前状态缓存

真正事实源建议优先级：

1. 事件流
2. 审计与追踪
3. 快照

## 11. 迁移审计要求

所有迁移动作建议写入审计：

- 谁触发了迁移
- 从哪个版本迁到哪个版本
- 迁移了哪些文件
- 是否发生回退
- 是否丢弃了未知字段

## 12. Bee 运行时与进化面的关系

- bee 运行时导出的评估输入必须声明版本
- 进化面应按版本解析，不得假设字段永远不变
- 注册表更新记录也应有独立 schema version

## 13. 第一版建议

第一版最少做到：

- 核心文件有 `schema_version`
- 快照写入原子化
- 损坏时支持 `.bak` 回退
- 支持从事件流重建最小任务状态
- 迁移动作进入审计

## 14. 总结

只要 Honeycomb 继续坚持文件系统优先，就必须把“版本、迁移、回退、重建”当成一等能力，而不是后补脚本。
