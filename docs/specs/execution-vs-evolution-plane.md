# 运行时、能力中心与进化中心

## 1. 为什么从双平面升级为三层

只拆“执行 vs 进化”不足以解决核心冲突：运行时智能执行与长期能力资产管理仍会挤在同一入口，导致二进制膨胀与职责混乱。

因此升级为三层：

- bee 运行时层：任务内智能执行
- hive 能力中心：全局能力资产管理
- ecology 进化中心：跨任务治理优化

## 2. Bee 运行时层（`honeycomb-bee`）

负责：

- queen/worker 运行
- LLM 驱动的任务内决策与执行
- 任务上下文、会话态、短期记忆管理
- 事件、追踪、审计证据回传

关键特征：

- 高频
- 任务内短周期
- 强实时
- 以完成当前任务为目标

## 3. Hive 能力中心（`honeycomb`）

负责：

- skill、tool、implementation、template、practice 的注册与发布
- 全局工具治理、能力约束与默认实现策略
- 对 bee 提供版本化能力视图

关键特征：

- 中频
- 长期资产管理
- 跨任务共享
- 以能力可复用与可治理为目标

## 4. Ecology 进化中心（`honeycomb-evolution`）

负责：

- 汇总全量运行观测
- 评分、谱系维护、晋升淘汰
- 最佳实践沉淀与治理决策执行

关键特征：

- 低频或批量
- 长期优化
- 跨任务统计与比较
- 以系统长期质量为目标

## 5. 三层数据关系

bee 输出：

- 任务结果
- 运行事件
- trace/audit
- 工具调用轨迹
- 评估输入

evolution 消费以上数据并输出：

- 评分报告
- 晋升/淘汰决策
- 谱系与实践更新建议

hive 消费治理决策并发布：

- 新默认实现
- 更新后的实践与能力配置

## 6. 强边界原则

- bee 不直接写长期能力注册表
- hive 不参与任务内实时派发循环
- evolution 不直接控制正在执行中的任务内决策
- 长期状态变更必须通过 evolution -> hive 的正式路径完成

## 7. 当前结论

1. Honeycomb 不再使用“双平面即可覆盖全部问题”的表述。
2. 运行时执行、能力管理、治理进化三层必须独立建模与独立演进。
3. LLM 核心执行能力归属 bee 层，长期能力治理归属 hive/evolution 层。
